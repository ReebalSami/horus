KIEval — **"Evaluation Metric for Document Key Information Extraction"** (arXiv 2503.05488, Mar 2025). An *application-centric* KIE metric that assesses models not only on individual-entity extraction but on **structured (grouped) information**, motivated by the gap between model-centric F1 and industrial correction-cost.

**Verified definitions (from the printed paper, per issue #74's "verify the grouping definition" mandate)**:

1. **Group-matching** (§4.1): predicted groups `PR = {pr₁…pr_N}` and ground-truth groups `GT = {gt₁…gt_M}`, each group a set of `(entity-type, value)` tuples. A matching score `S(n,m)` counts identical entities between `prₙ` and `gtₘ`; **Hungarian matching** pairs prediction↔GT groups, yielding `G` with `|G| = min(N, M)`. **Non-group entities** (e.g., `company.name`, `company.number`) are folded into a **1ˢᵗ group** representing same-structural-format entities.

2. **KIEval Entity F1** (§4.1): per matched group `(n,m)`, `TP/FN/FP` are counted per entity-type `e`; standard precision/recall/F1 over the pooled entity statistics.

3. **KIEval Group F1** (§4.1): evaluates **the entire group as one unit of information** — a group is TP only when the predicted and GT groups are **identical** (binary indicator `𝟙[·]`, all-or-nothing). Crucially, **group-level evaluation is conducted on all but the first element of G** (i.e. `G′`) because the first element represents the non-group entities.

4. **Grouping for text-generation models** (§5.3): "group-level information can be extracted by simple structuring of the target key-value pair text sequence such as JSON format strings." (Relevant because HORUS's cohort is generative VLMs.)

5. **KIEval_Aligned** (§4.2): an application-centric reformulation casting errors as correction costs (substitution = min(FP,FN); addition = remaining FN; deletion = remaining FP). HORUS adopts the **Group F1** lens (§4.1), not the Aligned cost metric — noted here for completeness.

**Role in HORUS (per ADR-027)**: HORUS adopts **KIEval Group F1** as metric #3 of the 4-metric scorer surface. The HORUS group partition over the 16-field registry:
- `seller` = {seller_name, seller_vat_id, seller_tax_id, seller_gln}
- `buyer` = {buyer_name, buyer_reference, buyer_vat_id}
- `totals` = {line_total_amount, tax_basis_total_amount, tax_total_amount, grand_total_amount, due_payable_amount}
- **non-group G′ (excluded)** = {invoice_number, issue_date, invoice_currency_code, delivery_date} — the document-level scalars, matching KIEval's "1ˢᵗ group / non-group entities" exclusion.

Since HORUS header invoices carry exactly one instance of each business group, Hungarian matching is trivially 1:1 (one seller group ↔ one seller group, etc.); the load-bearing KIEval contribution is the **all-or-nothing group-TP semantics** (a `totals` group scores only if ALL five money fields are correct), which surfaces whether a model extracts coherent blocks vs scattered fields.

**What HORUS does NOT take from this paper**: the KIEval_Aligned correction-cost metric (§4.2); the absolute numbers from the paper's CORD/SROIE experiments; the sequence-labeling slot-filling grouping (§5.3) — HORUS groups are a fixed declarative partition over a known EN16931 schema, not learned.
