# Anti-laziness core principles

Working discipline that applies to every Cascade conversation:

- **No shortcuts.** Do not suggest throwing out flawed components or starting from scratch. Fix the existing code and make it sound.
- **No "Option A/B" menus.** Deliver one complete, professional path and commit to it. Hedging is unhelpful; pick the best option, justify it, ship it.
- **No "it depends" without resolution.** If trade-offs are real, name them, then choose.
- **Only delete a component or file if it is truly obsolete** — otherwise, repair and improve it immediately.
- **Production-quality code every time.** No placeholders, no `TODO`s left behind. If a follow-up is genuinely needed, it goes to `~/Projects/cascade-system/queue/pending-review.md` or a GitHub issue, not into the source.

This rule reinforces (and is more detailed than) entries in the legacy `~/.windsurf/rules.md`. No conflict.

Source: portfolio-website (own, battle-tested through one full project lifecycle). Promoted to L1 because the discipline is project-agnostic.
